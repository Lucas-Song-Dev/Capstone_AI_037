# Verification Plan

## Purpose 

**Goal**: Use DRAMPower as a **reference simulator** to increase confidence that our DDR5 power model.

**Scope**
- DDR5 DIMM-level reporting where the configuration represents **two subchannels aggregated** (e.g., `x8` devices with `4 devices/subchannel` → `8 devices/rank` total).
- Comparing **core** and **interface** breakdowns between:
  - DRAMPower output (trace-based, per-command/state energy)
  - Our tool output (workload-parameterized, steady-state averages)

**Deliverables:**
1. Produces **reasonable per-component breakdowns** for both **core** and **interface** power at the **DIMM scope** (configured as **2 subchannels** aggregated).
2. Preserves **correct scaling trends** when key knobs change (e.g., `tCK`/data rate, device count, device width, and RD/WR intensity).



## Why Comparison is Valuable

Even if the models do not have identical internal definitions, DRAMPower comparison is valuable because it can reveal:

- **Scope errors**: per-device vs per-rank vs per-DIMM current tables; missing/extra ×(subchannels) or ×(devices) multipliers.
- **Incorrect trends**: e.g., dynamic I/O not scaling with frequency, refresh not scaling with `tRFC/tREFI`, RD/WR not scaling with access intensity.


## Reference Tool: DRAMPower (How We Use It)

DRAMPower is used as the reference because it computes energy from:

- A JEDEC-style memspec (timings, voltages, IDD/IPP tables)
- A command trace (cycle-accurate command/state timing)

### Build

- Build DRAMPower from `https://github.com/tukl-msd/DRAMPower`.

### Run (example)

From the DRAMPower repo root (example command used during development):

`./build/bin/cli -c build/bin/config.json -m tests/tests_drampower/resources/ddr5.json -t trace_1.csv -j out/output_trace_1_base.json`

Where:
- `-c` is the DRAMPower simulation config (data toggling, etc.). We treat this as “unmodified reference defaults” unless explicitly justified.
- `-m` is the DDR5 memspec JSON. To align with our tool, this file must reflect the same IDD/IPP/timing/geometry assumptions.
- `-t` is the input trace (e.g., generated by Ramulator). Document how the trace was generated and what it represents.
- `-j` is the output JSON containing the energy breakdown.

### Post-process / Report Average Power

We convert DRAMPower energy to average power using:

`python3 calculate_power.py --json out/output_trace_1_base_6400.json --cycles 2919445 --tck 0.312e-9 --memspec_json tests/tests_drampower/resources/ddr5.json`

Where:
- `--json` is the DRAMPower output JSON (`-j` above).
- `--cycles` is total simulated cycles covered by the trace.
- `--tck` is the DRAM clock period used to compute real time (`T = cycles * tCK`).
- `--memspec_json` is optional fallback to locate `tCK` when not embedded.

## Alignment

For each comparison run, record and align:

- Geometry/topology: ranks, bank groups, banks, rows/cols; **2 subchannels aggregated**; device width; devices per subchannel/rank.
- Timing: `tCK`, `tRCD`, `tRAS`, `tRP`, `tRFC`, `tREFI` (and any other timings used by either tool).
- Voltages: `VDD`, `VDDQ`, `VPP`.
- IDD/IPP current tables: confirm whether the table is **per DRAM device** (JEDEC IDD-style) and not module-aggregated.
- Trace/window definition (DRAMPower): cycles count and `tCK` used to compute average power.

### Required Scaling Sweeps

1. **`tCK` (data rate) sweep**
   - Goal: confirm `if.dram_dynamic` / `P_total_interface_dyn` increases ~linearly with frequency (all else equal).
   - Report: `P_total_interface_dyn` vs `data_rate_MTps`.

2. **Device width / device count sweep (DIMM-equivalent)**
   - Compare x8 vs x16 while holding DIMM bus width constant.
   - Goal: ensure we do not implicitly assume per-device IDD is unchanged across widths; use correct memspecs per width.
   - Report: `P_core` vs `(width, devices_per_rank)` for both tools.

3. **Read scaling sweep**
   - Increase read intensity (e.g., 0.25×, 0.5×, 1×, 2×) while holding everything else fixed.
   - Expectation: `core.rd` scales approximately linearly with read intensity; interface dynamic also increases.


## Problems Encountered
some project memspecs (e.g., `core/workloads/micron_16gb_ddr5_4800_x8_spec.json`) may reflect **module/DIMM-level** assumptions rather than **per-device** JEDEC IDD/IPP tables. Our core model assumes IDD/IPP are **per DRAM device**, and DIMM power is computed by multiplying by the number of modeled devices.
- **Symptom**: when we feed our model the same per-device-like spec used for DRAMPower, our tool can report materially lower power (observed on the order of ~40% in prior experiments).
